# RRaM - техническое ТЗ

## 1. Цель

Создать онлайн-версию RTS/RPG игры RRaM с картой, персонажами, бордами, кубиками, карточной системой и возможностью сетевой игры.

## 2. Базовые сущности

### 2.1 Игрок

Поля:

- `id`
- `name`
- `team`
- `is_active`
- `remaining_rolls`
- `characters`
- `hand` или общий набор карт игрока, если карты хранятся не только у персонажей.

### 2.2 Персонаж

Поля:

- `id`
- `owner_id`
- `type`
- `position`
- `inventory`
- `is_alive`

Типы персонажей:

- `blacksmith`
- `blacksmith_assistant`
- `warrior`
- `hunter`
- `shaman`

### 2.3 Борда

Поля:

- `id`
- `q`
- `r`
- `terrain_type`
- `is_start_point`
- `start_point_type`
- `occupants`

Для ромбообразной сетки можно использовать координаты `q`, `r`.

### 2.4 Карта

Поля:

- `id`
- `name`
- `type`
- `owner_id`
- `holder_character_id`
- `effect`
- `is_consumable`

### 2.5 Бросок

Поля:

- `id`
- `player_id`
- `die_1`
- `die_2`
- `used_for`
- `created_at`

## 3. Основные действия

- Начать игру.
- Расставить персонажей по стартовым точкам.
- Выполнить бросок двух кубиков.
- Передвинуть персонажа.
- Передать карты между персонажами.
- Добрать карту из колоды.
- Использовать карту.
- Телепортировать персонажа.
- Завершить ход.

## 4. Серверные проверки

Сервер должен проверять:

- Доступность хода игрока.
- Корректность значения кубиков.
- Достаточность оставшихся бросков.
- Возможность перемещения по бордам.
- Наличие карт у передающего персонажа.
- Лимит передаваемых карт по значению кубика.
- Право взять карту из выбранной колоды.
- Право телепортации на выбранную подписанную точку.

## 5. Открытые технические решения

- Формат игры: пошаговая, real-time with turns или гибрид.
- Количество игроков в матче.
- Формат карты: JSON, tiled map, hex/grid data или custom editor.
- Клиент: web, desktop или mobile.
- Сервер: authoritative server обязателен для честной онлайн-игры.
- Нужна ли синхронизация в реальном времени через WebSocket.
