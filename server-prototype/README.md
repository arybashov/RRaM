# RRaM Server Prototype

Минимальный каркас будущего сервера для онлайн-версии RRaM. Эта папка изолирована от `prototype-web/` и не меняет текущий веб-прототип.

## Назначение

Серверный прототип нужен, чтобы постепенно перенести RRaM от локального/визуального прототипа к сетевой игре с авторитетным сервером.

На первом этапе сервер должен:

- принимать WebSocket-подключения игроков;
- создавать игровые комнаты;
- хранить минимальное состояние комнаты на сервере;
- принимать команды игроков;
- рассылать события всем участникам комнаты;
- оставлять спорные игровые правила в отдельных модулях, чтобы их можно было менять без переписывания транспорта.

## Текущий состав

- `package.json` - описание Node-пакета и скрипты запуска.
- `src/index.js` - точка входа Fastify-сервера.
- `src/game-state.js` - минимальная модель комнат и игроков.
- `src/protocol.js` - названия клиентских команд и серверных событий.

## План MVP сервера

1. Подключение клиента по WebSocket.
2. Создание комнаты через команду `room:create`.
3. Вход в комнату через команду `room:join`.
4. Рассылка снимка состояния комнаты всем участникам.
5. Черновая команда `turn:roll`, которая фиксирует бросок кубиков на сервере.
6. Отдельное описание правил перемещения, карт и телепортации перед переносом их в код.

## Предлагаемый стек

- Node.js 20+
- Fastify
- `@fastify/websocket`

Зависимости пока не установлены. После подтверждения стека можно будет выполнить:

```bash
npm install
```

## Запуск после установки зависимостей

```bash
npm run dev
```

По умолчанию сервер слушает порт `8787`.

Проверка HTTP:

```bash
curl http://localhost:8787/health
```

WebSocket endpoint:

```text
ws://localhost:8787/ws
```

## Формат сообщений

Клиент отправляет JSON:

```json
{
  "type": "room:create",
  "payload": {
    "playerName": "Игрок 1"
  }
}
```

Сервер отвечает JSON-событиями:

```json
{
  "type": "room:snapshot",
  "payload": {
    "room": {
      "id": "abc123",
      "players": []
    }
  }
}
```

## Принципы для следующей итерации

- Сервер должен быть авторитетным: клиент запрашивает действие, сервер проверяет и применяет.
- WebSocket-события не должны содержать скрытую информацию, недоступную конкретному игроку.
- Игровые правила нужно держать отдельно от сетевого транспорта.
- Каждое изменение комнаты должно иметь `revision`, чтобы клиент мог безопасно синхронизироваться.
- Для раннего прототипа состояние можно хранить в памяти процесса; базу данных добавлять позже.

- Как именно задается карта: координаты бордов, острова, стартовые точки?
- Должен ли сервер сам бросать кубики или принимать результат из внешнего источника?
- Какой формат карт действий нужен для первой играбельной версии?
