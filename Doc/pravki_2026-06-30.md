# Правки в игру от 2026-06-30

Цель: зафиксировать, что нужно внести в игру по картам и колодам.

Истина для переноса:

1. Последний `rram-deck-overlay-project.json`.
2. Фактические разделители колод из `deckSeparators`.
3. Последние уточнения от 2026-06-30.
4. Документ `Doc/Применение к противнику.docx` как источник уточнений по спорным картам.

Код игры сейчас не считаем источником истины. Его нужно привести к JSON и последним уточнениям.

## Главное правило по колодам

Колоды в игре нужно пересобрать заново по текущему JSON.

Фактическая колода карты определяется не старым `deck`/registry-полем, а местом карты между разделителями `deckSeparators`.

Например:

- `Иерихон` в карточном поле может иметь `deck: red`, но фактически по разделителю лежит в `Чертежи`.
- `Задание на молот Иерихон` может иметь смешанные старые registry-deck значения, но фактически лежит в `Чертежи`.
- Такие старые поля нельзя использовать как источник состава колод.

## Профессиональные колоды

- Чертежи и рецепты берутся только через свои кнопки.
- Кнопка `Чертежи` работает с колодой `Чертежи`.
- Кнопка `Рецепты` работает с колодой `Рецепты`.
- Обычные клетки добычи/событий не должны тянуть карты из профессиональных колод.
- Стартовые базовые карты из раздела `Базовые карты` остаются по JSON, но дальнейший добор чертежей/рецептов идет через профессиональные кнопки.

## Карта-лицо и карта-экземпляр

Одна карта-лицо может встречаться в разных колодах/рубашках.

Нужно разделить:

- `faceId` / `cardId` - что это за карта по лицу и правилам;
- `instanceId` - конкретный экземпляр карты;
- `sourceDeck` - из какой колоды взят конкретный экземпляр;
- `sourceBack` - какая рубашка у конкретного экземпляра.

Правило возврата:

```text
"в свою колоду" / "в соответствующую колоду" = вернуть конкретный экземпляр в его sourceDeck/sourceBack.
```

Нельзя возвращать карту только по `cardId`, потому что одно лицо может быть в разных колодах.

## Состояния карты

Нужно поддержать минимум 3 разных состояния:

- `рубашкой вверх` = скрытая карта, лежит рубашкой к игроку;
- `описанием вверх` = карта открыта лицом/текстом к игроку;
- `вверх ногами` = карта открыта лицом/текстом к игроку, но повернута на 180 градусов.

Важно:

- `вверх ногами` не равно `рубашкой вверх`;
- `вверх ногами` не равно обычному перевороту лицом вниз;
- это отдельное состояние для ловушек, ядов и похожих эффектов.

## Мутация карты

Нужна отдельная механика мутации карты.

Пример: превращение зверя/персонажа в лягушку.

Это не переворот карты и не смена рубашки. Это временное или постоянное состояние эффекта:

- исходная карта остается известной;
- поверх нее применяется состояние `mutated`;
- у мутации есть тип, например `frog`;
- правила атаки/оружия/защиты читаются с учетом мутации.

## Правило "палаток или лучше"

`Палаток` = базовый молоток кузнеца, стартовая/базовая карта кузнеца.

Категория `палаток или лучше`:

- `hammer` Палаток / Молоток;
- `art_dark_forest_020` Деревянный молоток;
- `irikon` Иерихон.

Если в чертеже написано `положите палаток или лучше`, игра должна принимать любой из этих вариантов.

Дополнительно из `Doc/Применение к противнику.docx`:

- Палаток можно использовать как оружие против всех, кроме зверей: 15 урона.
- Деревянный молоток при изготовлении чертежа делает предмет с первого раза, если все условия удовлетворены.
- Деревянный молоток можно использовать как оружие против всех, кроме зверей: 25 урона.
- Иерихон при изготовлении чертежа позволяет изготовить любой предмет при любых выпавших кубиках, если есть все ингредиенты.

## Правки конкретных карт

### Мраморный самоцвет, `art_lake_001`

Текущее описание в JSON содержит старую/спорную механику и `???`.

В игру внести последнее уточнение:

```text
Если положить эту карту в драке, то весь ваш урон умножается на 2.
Если карта была положена в драке с одним персонажем, а потом в драку подключились еще враги, ваш урон все равно остается удвоенным.
Можно использовать каждый 4-й раз сброса кубиков.
```

Требования к механике:

- хранить счетчик сбросов кубиков для условия `каждый 4-й раз`;
- эффект удвоения урона сохраняется при подключении новых врагов в текущую драку.

### Чертеж на легкую кольчугу, `art_dark_forest_005`

Текущее описание в JSON содержит `????`.

В игру внести:

```text
После выполнения условий чертеж открывается.
Карта уже есть и работает.
```

Дополнительно:

- `палаток или лучше` принимает Палаток, Деревянный молоток или Иерихон;
- после успеха кузнец открывает чертеж и достает изделие.

### Жест / Жесто, `armor_zhest`

Текущее описание в JSON содержит неизвестное число:

```text
суммарное количество кубиков не менее ?
```

В игру внести:

```text
Если применяете Жесто к противнику, он не может атаковать, но его броня увеличивается на 15.
Чтобы снять заклятье Жесто, противнику нужно кинуть суммарное количество кубиков 6+.
Жесто как броню можно использовать каждую драку.
Жесто как заклятье можно использовать каждый 3-й сброс кубиков.
```

Требования к механике:

- режим брони и режим заклятья считаются разными применениями;
- броня доступна каждую драку;
- заклятье доступно каждый 3-й сброс кубиков;
- пока заклятье активно, цель не атакует, но получает +15 брони;
- снятие заклятья: сумма кубиков 6+.

### Иерихон, `irikon`

В JSON находится в фактическом разделе `Чертежи`.

В игру внести:

- относится к категории `палаток или лучше`;
- используется как квестовый инструмент/молот;
- квест связан с Фениксом и золотым пером;
- при использовании как оружие/молот работает по описанию карты;
- если используется для воскрешения: передать шаману, выложить Иерихон и ковер шамана, кинуть сумму двух кубиков 5+, выбрать аватарку мертвого союзника, воскресить союзника на соседней клетке от шамана.

### Задание на молот Иерихон, `task_irikon`

В JSON находится в фактическом разделе `Чертежи`.

В игру внести:

- это не карта обычной колоды Феникса;
- это профессиональная карта/задание из `Чертежи`;
- условие выполняется через квест Феникса и золотое перо;
- драться с Фениксом может кто угодно, но задание/чертеж должен быть у кузнеца в инвентаре.

### Золотое перо к кузнецу противника, `art_fairy_glade_003`

Работает так же, как золотое перо к своему кузнецу, только цель доставки - камень кузнеца врага.

### Феникс, `phoenix_1`

Игровой механизм Феникса не менять.

Нужно только учитывать связку:

- Феникс;
- золотое перо;
- задание/чертеж на Иерихон;
- доставка пера к нужному камню кузнеца.

## Описания карт

Все описания карт брать из последнего JSON, кроме правок, явно перечисленных в этом документе.

Если в JSON есть старые `???`, `????` или `?`, для перечисленных карт применять последние уточнения из этого документа.

## Что убрать из старой логики игры

- Не использовать старые составы колод из кода как истину.
- Не определять колоду карты только по старому `deck`/registry-полю.
- Не сливать одинаковые лица в одну общую колоду без учета рубашки/экземпляра.
- Не считать `вверх ногами` обычным `faceDown`.
- Не считать мутацию карты простым переворотом.

## Что добавить в игру

- Генерацию/загрузку колод по текущему JSON и разделителям.
- Хранение `sourceDeck` и `sourceBack` у каждого экземпляра.
- Возврат карт в исходную колоду конкретного экземпляра.
- Отдельное состояние `upsideDown` для открытой карты, повернутой на 180 градусов.
- Механику мутации карты, например `frog`.
